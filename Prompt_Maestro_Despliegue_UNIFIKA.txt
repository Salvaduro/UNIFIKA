1. ESPECIFICACIÓN DEL STACK TECNOLÓGICO Y HERRAMIENTAS
--------------------------------------------------------------------------------
- Backend: Python 3.13 con FastAPI (Uvicorn como servidor ASGI).
- Frontend: React (Vite) con Tailwind CSS para estilos de interfaz.
- Base de Datos: PostgreSQL alojado en Supabase (Lógica de sincronización vía 'ON CONFLICT DO UPDATE SET').
- Fuente de Datos Maestra: CRM Wolkvox (Consumido vía API para sincronización de empleados y novedades).
- Generación de Documentos: ReportLab en Python para el renderizado dinámico de PDFs de desprendibles de pago.
- Control de Versiones: Git (La última versión estable de esta fase está sellada bajo el tag 'v1.0-MVP').

--------------------------------------------------------------------------------
2. RESUMEN DE AVANCES Y LOGROS CONSOLIDADOS (HITO MVP COMPLETO)
--------------------------------------------------------------------------------
A. Motor Matemático de Nómina:
   - Extracción de datos crudos del CRM Wolkvox y liquidación exacta de nómina colombiana (salarios, horas extras, recargos nocturnos, dominicales/festivos, incapacidades, vacaciones y auxilios).
   - Regla Crítica para Tiempo Parcial (Decreto 2616 de 2013): 
     * Las horas extras y recargos se excluyen del IBC de seguridad social (Salud, Pensión, ARL, Caja) para evitar distorsiones del modelo de cotización por días de la UGPP.
     * Excepción ya codificada: Si el empleado de Tiempo Parcial tiene activa la opción de pagar salud (asumida por el empleador), el IBC se fuerza automáticamente al 100% de 1 SMLV (vigente en Colombia para el año en curso), sin sumarle horas extras.

B. Bóveda de Datos Relacional (Supabase):
   - Tablas maestras e históricas: 'm_aportantes', 'm_empleados' (que ahora incluye la columna 'link_drive' para enlaces del expediente histórico), 't_novedades', y 't_cierres_nomina'.
   - Llave Compuesta de Temporalidad en 't_novedades': 'id_contrato' + 'periodo_liq' + 'quincena_pago'.

C. Motor de PDFs ReportLab:
   - Generación exacta del comprobante con nomenclatura inteligente (ej. 'Desprendible_ID_JUNIO_2026_Q1.pdf') listo para descarga.

D. Blindajes de Seguridad y UX:
   - Bloqueo de Nómina Cerrada: Validación en backend que consulta 't_cierres_nomina' antes de guardar. Si el periodo está cerrado, lanza un HTTP 400 con un mensaje específico indicando que la nómina se encuentra cerrada y que deben contactar al canal de atención.
   - El frontend captura este error HTTP 400 y lo proyecta en un banner/toast que dura exactamente 5 segundos (5000ms).
   - Clean Code: Limpieza total de 'print()' en Backend (bloques except huérfanos parchados con 'pass' o 'logging') y eliminación de 'console.log()' o imports muertos en Frontend.

4. NUEVA MISIÓN: CAMINO A - DESPLIEGUE A PRODUCCIÓN (CLOUD DEPLOYMENT)
--------------------------------------------------------------------------------
Nuestro objetivo inmediato en este cuaderno es llevar la aplicación de un entorno local a una infraestructura en la nube real. Para ello, implementaremos el siguiente plan por fases:

Fase 1: Preparación del Backend (FastAPI) para Render o Railway:
   - Configuración de CORS para aceptar únicamente el dominio de producción del Frontend.
   - Migración de la conexión de base de datos local a variables de entorno dinámicas.
   - Configuración del comando de arranque en producción usando gunicorn con uvicorn workers o uvicorn estándar (`uvicorn main:app --host 0.0.0.0 --port $PORT`).
   - Asegurar que las llamadas externas al CRM Wolkvox y Supabase consuman las variables de entorno inyectadas de forma segura.

Fase 2: Preparación del Frontend (React + Vite) para Vercel:
   - Configuración de las variables de entorno de Vite en producción (ej. 'VITE_API_BASE_URL' apuntando al backend en producción de Render/Railway).
   - Asegurar que la compilación de producción (`npm run build`) no arroje errores de Types o dependencias y se configure correctamente el archivo 'vercel.json' para el manejo de rutas SPA (Single Page Application).

Fase 3: Configuración de Bases de Datos en Supabase:
   - Asegurar que las variables de conexión apunten al entorno seguro de producción.
   - Validar que los triggers de fecha y hora usen estrictamente la estampa de tiempo parametrizada 'America/Bogota'.
